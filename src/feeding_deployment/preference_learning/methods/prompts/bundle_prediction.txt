IMPORTANT — PREFERENCES ARE CORRELATED: the user's preferences co-vary through latent factors that are not directly observed: the user's current state (e.g., hurried, fatigued, relaxed), the physical conditions of this meal (setting, time of day, lighting and similar environmental factors), and properties of the meal itself. Each CORRECTION is a direct observation of what those latent factors are doing right now. A dimension does NOT need its own correction to change: when the user corrects one dimension, infer which latent factor most plausibly produced that correction, and update every OPEN dimension that same factor influences — categorical and continuous alike. Do not fall back on "this dimension was not itself corrected, so keep its previous value": a seed or memory value carried over from previous meals cannot reflect anything the user revealed DURING this meal, so where a this-meal correction bears on a dimension — even indirectly, through a shared latent factor — it outweighs the carried-over value. Similar prior meals in EPISODIC MEMORY show what the user actually preferred when those latent factors were in a comparable state; weigh them accordingly. Confirmations are weaker evidence: they tell you the prediction was acceptable, not that the latent factors changed.

YOUR OBJECTIVE is to minimize the number of corrections the user still has to make this meal. Every open dimension is still presented to the user before it takes effect, so an updated prediction that turns out wrong costs the user nothing extra — they correct it exactly as they would have corrected the stale value — while keeping a carried-over value that this meal's evidence already contradicts guarantees one more correction. Never hold a value "pending the user's own correction" when a correction already made this meal, combined with a shared latent factor, tells you what the user is going to ask for.
